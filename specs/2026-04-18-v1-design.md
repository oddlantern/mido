# Neutron v1.0 — Design Spec

**Date:** 2026-04-18
**Status:** Approved for implementation (pending user review of written spec)
**Supersedes:** phase list in `memory/project_roadmap.md`

## Executive summary

Neutron 1.0 is a stability pledge delivered through a stair-step of five minor releases. Each minor is independently useful; 1.0 ships on top of the final minor as a pure version bump + API freeze. Scope: hardening floor (test coverage + ecosystem parity) plus two new capabilities (task runner with replay cache, glob package paths). Deferred to post-1.0: remote cache, CI configurability.

## Current state (baseline: 2026-04-18)

- Version: 0.2.2, published as `@oddlantern/neutron`
- Release automation: release-please firing on every merge; CI runs typecheck + lint + format-check + build + test + publint on every PR
- Plugin API declared stable (0.2.2 / commit `c3c5b20`)
- External plugin loading shipped (0.2.0)
- Six ecosystem plugins present: TypeScript + Dart complete; Python/Rust/Go/PHP flagged `experimental: true`
- Codebase: ~117-file build output, 932 tests across 87 files

## Decision log

Choices locked during brainstorming:

| Decision | Choice | Rationale |
|---|---|---|
| v1 bucket | Hybrid (hardening + task runner + globs) | Task runner is the feature neutron needs to credibly call itself a workspace tool; coverage is the gate for a stability pledge; remote cache / CI configurability are post-1.0 polish |
| Task runner scope | Task DAG with cross-task deps | Turborepo/Nx model; minimal (`^task` only) breaks on real workflows |
| Cache semantics | Replay outputs + logs on cache hit | Skip-only creates footguns when `dist/` is cleaned; every serious tool ended up here |
| Release cadence | Incremental stair-step minors, 1.0 as pure version bump | Release-please already fires per-merge; fighting that rhythm has no benefit |
| Parity scope | All 5 tiers in one milestone | Auditable exit criteria; deferring tiers leads to "forever experimental" ecosystems |
| Parity sequencing | Width-first (Tier 1 across all ecosystems, then Tier 2, …) | Users see progress across the whole matrix; cross-ecosystem consistency easier to enforce |
| Coverage target | ≥90% | Not vanity; the gate for the 1.0 stability pledge |

## Milestone sequence

Six releases. Release-please picks the actual version numbers; the labels below are planning handles.

```
0.2.2 (now) → 0.3 → 0.4 → 0.5 → 0.6 → 0.7 → 1.0
```

### 0.3 — Glob package paths

**Scope:** `packages:` field accepts globs (`"packages/*"`, `"apps/**"`, `"!experimental/*"`, `"{tools,scripts}/*"`).

**Why first:** smallest change; unblocks anyone with more than a handful of packages; does not depend on anything downstream.

**Invariants preserved:**
- Literal paths are a valid subset of the glob grammar — existing configs need zero changes.
- Expansion happens at config load; graph builder never sees a glob.
- Dedup after expansion; alphabetical order (predictable logs and diffs).
- Empty match per pattern: warn (not error). Zero packages total: error.

**Implementation:** single change in `src/config/schema.ts` + `src/config/loader.ts`. `tinyglobby` already in dependency graph — no new runtime dep.

### 0.4 — Task runner foundation (no cache yet)

**Scope:** `tasks:` field in `neutron.yml`, `neutron run <task>`, task DAG with Turborepo-style `^task` / `task` / `^:*` / `:*` syntax, `neutron task` CLI lifecycle (`init`, `add`, `ls`, `validate`), `--affected` filtering built into `run`.

**Why before cache:** the DAG surface is the hard design; hot execution proves the shape before hashing gets baked in. Ships a few weeks of dogfood time before cache semantics freeze.

### 0.5 — Cache layer

**Scope:** replay cache (outputs archive + stdout/stderr logs) keyed on `hash(inputs + resolved command + declared env + neutron version + plugin version)`; `--force` bypass; `neutron cache ls` / `neutron cache clean`.

**Why after 0.4:** cache correctness depends on task semantics being proven in production use.

### 0.6 — Ecosystem parity (Python / Rust / Go / PHP)

**Scope:** all 5 tiers from the plugin parity audit. See *Hardening — parity* section below.

**Exit criteria for removing `experimental: true`:** Tier 1 + Tier 2 complete for the ecosystem, ≥2 framework adapters OR documented "no adapters, use `entryFile` override".

### 0.7 — Coverage push

**Scope:** push source coverage to ≥90% via Bun's built-in coverage; add integration tests through the built binary for every mutating command; compile-check codegen outputs.

**Why last:** covering code that's about to grow is waste. Finish growing (parity), then cover.

### 1.0 — Stability pledge

No new features. Version bump + CHANGELOG narrative section + badge swap + announcement. API surface frozen per the pledge below.

## Task runner architecture

### Config shape

```yaml
# neutron.yml
tasks:
  build:
    dependsOn: ["^build"]                 # ^ = same task in upstream local deps
    inputs: ["src/**", "package.json"]
    outputs: ["dist/**"]
    # no `command:` → delegates to ecosystem plugin's BUILD action per package

  test:
    dependsOn: ["build"]                  # local (same-package) task dep
    inputs: ["src/**", "test/**"]

  deploy:
    dependsOn: ["^:*"]                    # all upstream tasks in upstream deps
    command: ./scripts/deploy.sh          # explicit shell, bypasses plugin
    cache: false                          # side-effect task, no caching
    env: ["AWS_REGION", "DEPLOY_ENV"]     # env captured into cache key
```

### Dependency grammar

- `<task>` — same-package task
- `^<task>` — same task in upstream local deps (Turborepo default)
- `^:*` — any task in upstream local deps must complete first
- `:*` — any task in same package (rarely useful; included for symmetry)

### Auto-delegation rule

When a task entry has no `command:`, dispatch is deterministic:

```
For task <T> in package P:
  plugin = resolvePlugin(P.ecosystem)
  actions = await plugin.getActions(P, root)
  if <T> in actions → await plugin.execute(<T>, P, context)
  else             → skip P with note "task <T> not supported by <ecosystem> plugin"

At config load time:
  if no package in the workspace supports <T> → error
```

Three honest behaviors fall out:

- **Standard action match** (`tasks.build` → TS plugin's `BUILD`, Rust's `BUILD`, Python's nothing yet → skipped until 0.6 Tier 2 lands Python `BUILD`)
- **Custom action match** (`tasks.generate-openapi-ts` → TS plugin's custom action; no-op elsewhere)
- **Typo** (`tasks.buld`) → hard error at load time

The "skip with note" path makes gradual rollout sane: when Python has no `BUILD` yet, Python packages get visibly skipped rather than failing mysteriously.

**Caveat:** plugin `getActions()` must honestly reflect what it supports. Currently true; becomes load-bearing rather than advisory. Reflected in the 0.6 audit work.

### DAG execution (0.4)

- Task graph = (package topological order from `src/graph/topo.ts`, which already exists) × (task deps declared in config).
- Parallel execution within each DAG wave; sequential across waves.
- `--parallel=N` caps concurrency (default: CPU count).
- Fail-fast: failed task prunes its downstream; siblings finish; overall exit non-zero.

### Cache key (0.5)

```
hash(
  input file contents (glob-expanded per task),
  resolved command string (post-plugin dispatch),
  declared env vars (read, not all process env),
  neutron version,
  matched plugin version
)
```

Plugin version inclusion: a plugin upgrade invalidates the cache. Otherwise a codegen-change in the TS plugin would silently use stale cached outputs.

### Cache layout and hit behavior (0.5)

```
.neutron/cache/
  <cache-key>/
    outputs.tar.zst      # declared `outputs:` archived
    stdout.log
    stderr.log
    manifest.json        # task name, pkg path, timestamps, duration, schema version
```

- **Hit** → extract archive over package dir, replay stdout/stderr to current terminal, skip execution.
- **Miss** → execute, capture, write archive.
- `--force` bypasses cache read (still writes).
- `neutron cache clean` clears entire cache.
- `neutron cache ls` lists by size / age.

### CLI surface (0.4 + 0.5)

```
neutron run <task>                     # all packages, topological
neutron run <task> --affected          # affected by git diff + DAG closure
neutron run <task> --filter=<pkg>      # single package + upstream deps
neutron run <task> --parallel=4        # concurrency cap
neutron run <task> --force             # bypass cache read
neutron run <task> --dry-run           # print DAG, don't execute

neutron task init                      # scan packages, propose starter tasks:, write to neutron.yml
neutron task add <name>                # interactive: dependsOn, inputs, outputs, command
neutron task ls                        # per-package resolution table — debug auto-delegation
neutron task validate                  # DAG cycles, unknown actions, unused inputs (runs inside neutron check)

neutron cache ls                       # 0.5
neutron cache clean                    # 0.5
```

### Interaction with existing commands

- `neutron build`, `neutron lint`, `neutron fmt`, `neutron test`, `neutron generate` **remain** — we explicitly rejected the "subsume into `run`" option (would disrupt the stable plugin API).
- They become thin wrappers: if `neutron.yml` has a `tasks:` entry matching the name, delegate to the runner; else fall through to today's direct plugin dispatch.
- **Zero behavior change** for users who never add a `tasks:` block.
- **Affected-everywhere for free**: once `neutron build` delegates to the runner, `--affected` works across all commands — closes the "affected filtering for all commands" gap from the competitive analysis.

### `neutron dev` integration

Already watches `neutron.yml` with a 500ms debounce (`CONFIG_FILENAME` constant at `src/watcher/dev.ts:32`). Task runner inherits this:

- Edit `tasks:` during `dev` → DAG rebuilds, cache invalidates for affected tasks, in-flight tasks on removed definitions cancel cleanly.
- Task definition is part of the cache key: changing `inputs:` invalidates cached entries automatically.

### Non-goals for the task runner

- `^^` transitive task deps (Turborepo's obscure feature) — `^task` covers 99% of cases.
- Remote cache — post-1.0.
- Task-scoped plugins — plugins own `STANDARD_ACTIONS` and codegen pipelines; tasks are user-owned.
- Plugin-defined task pipelines beyond today's bridge pipelines — 2.0 architectural question.
- `neutron task rm` — deleting a YAML key is not worth a command.

## Hardening — parity (0.6)

### Baseline (from plugin parity audit)

TS and Dart plugins set the bar. Python/Rust/Go/PHP ship with meaningful gaps documented in `memory/reference_plugin_parity_audit.md`. Full capability matrix lives there; summary below.

### Five tiers, shipped width-first

Each tier runs across all four experimental ecosystems before the next tier begins.

**Tier 1 — Cross-command correctness** (cheap, unblocks `dev` / pre-commit):
- Add `suggestWatchPaths` to Python, Rust, Go, PHP
- Wire `resolvedFiles` through lint/format execution in all four
- Finish `context.lint*` / `format*` plumbing for Rust, Go, PHP

**Tier 2 — Missing standard actions** (honest parity):
- Python: BUILD (`python -m build` or PDM/Poetry equivalent)
- PHP: BUILD (`composer install --no-dev --optimize-autoloader`), TYPECHECK
- Go: LINT_FIX (`golangci-lint run --fix`)
- PHP: LINT_FIX (phpstan baseline OR overlap with format)

**Tier 3 — Tool resolution depth**:
- Rust: rustup toolchain detection, `+channel` support
- Go: GOPATH/bin + GOTOOLDIR before PATH
- Python: Poetry/PDM venv conventions

**Tier 4 — Framework adapters** (one additional per ecosystem):
- Python: Django-Ninja OR Flask-Smorest
- Rust: actix-web OR rocket
- Go: gin OR echo
- PHP: Laravel (Scribe OR L5-Swagger)

**Tier 5 — Content gaps**:
- Assets codegen for Python/Rust/Go/PHP (OR explicit "not applicable" declaration)
- Manifest-script discovery in `getActions` for each ecosystem (`[tool.pdm.scripts]`, `composer.json.scripts`, cargo-make, `go:generate`)

### `doctor` command growth (cross-cutting)

Extended during 0.6 to:
- Check toolchain for each present ecosystem (cargo, python3, composer, go)
- Report framework-adapter coverage for detected server packages
- Report cache health (`.neutron/cache/` size, oldest entry) after 0.5 ships

## Hardening — coverage (0.7)

### Target: ≥90% source coverage

Not vanity; the gate for 1.0's stability pledge. Rationale: the coverage phase runs *after* parity closes so we cover the final surface, not a moving one.

### Tool choice: Bun built-in coverage (`bun test --coverage`)

- Zero config cost; already the test runner.
- No new runtime dep.
- Same coverage output local + CI.

### Coverage priority order

1. `src/commands/upgrade.ts` — atomic manifest + lock writes
2. `src/commands/rename.ts` — `assertWithinRoot` path-traversal guard
3. `src/outdated/level3.ts` — live validation in temp dir
4. `src/plugins/builtin/ecosystem/{python,rust,go,php}/plugin.ts` — ecosystems completed in 0.6
5. `src/plugins/builtin/domain/schema/plugin.ts` — newly-added domain
6. Task runner + cache (0.4 + 0.5 code)

### Test-type policy

- **Unit tests** for pure functions (parsers, graph, hash computation, config schema).
- **Integration tests via built binary** (`dist/bin.js`) against `mkdtempSync` workspaces for any command that mutates files or shells out. Non-negotiable for `rename`, `install`, `upgrade`, `run`, `task add`.
- **Codegen tests compile the output** (`python -m py_compile`, `cargo check`, `dart analyze`). Substring assertions alone are insufficient — a syntax regression must fail a test.
- **`test.skipIf(!whichTool())`** on toolchain-dependent tests; CI installs each toolchain; local dev skips gracefully.
- **Both paths, always.** Every check function has a pass-case test AND a fail-case test.

### CI integration

Coverage artifact on every PR. Release-please release-body includes coverage delta. Hard gate for 1.0 cut: ≥90% source (excluding `dist/`, `test/fixture*`, `scripts/`).

## 1.0 stability pledge

### In scope (stable — breaking change ⇒ 2.0)

| Surface | Scope |
|---|---|
| CLI commands + flags | `neutron run`, `neutron task`, every existing command. Flag names, default values, argument shapes. |
| `neutron.yml` schema | Every top-level field, nested field, enum value. Config migrations handle old shapes forward. |
| Plugin API | `EcosystemPlugin`, `DomainPlugin`, `ExecutionContext`, `STANDARD_ACTIONS`, `DomainCapability`. Already declared stable in 0.2.2 — 1.0 enforces it. |
| Library subpath exports | `@oddlantern/neutron`, `/config`, `/graph`, `/parsers`, `/plugins`, `/checks`. Named exports; no new required parameters. |
| Exit codes | `0` success, `1` generic failure, documented specific codes per command. |
| `--json` output format | JSON shape of `graph --json`, `outdated --json`, `check --json`. |

### Out of scope (internal — can change any time)

- Internal function signatures, type exports outside public subpaths, `dist/` bundle layout
- `.neutron/cache/` directory layout (versioned internally via `manifest.json` schema tag)
- Human-readable log output format (text, colors, spinners) — only `--json` is stable
- Bundled oxlint/oxfmt versions — they can bump majors without a neutron major
- Generated `schema.json` file shape — derived from Zod; Zod schema is stable, file shape is not

### Deprecation policy

- No breaking change without deprecation between minors.
- Deprecation warning → removal: **minimum 2 minor releases** or ~3 months (whichever is longer).
- `NEUTRON_DEPRECATION_WARNINGS=error` env var for CI pipelines to fail on any deprecated-API use.
- Deprecated APIs emit a one-line warning with exact replacement and removal version.

### 1.0 release itself

- No new features in the 1.0 commit. Same tree as 0.7.x with:
  - Version bump (release-please configured to allow manual major)
  - CHANGELOG summary: *"What 1.0 means"*
  - README badge swap
- Announcement is separate from the code release.

## Non-goals for 1.0

Writing these down so scope creep during the stair-step has something concrete to bump against.

### Features deliberately deferred

| Not in 1.0 | Why | Post-1.0 home |
|---|---|---|
| **Remote cache** | Local cache proves the model first; remote has auth + storage + trust concerns | 1.x minor |
| **CI configurability** (dynamic workflow generation) | Static workflow works for first-party use; dynamic is large for a minority | 1.x minor or plugin |
| **`^^` transitive task deps** | Obscure; `^task` covers 99% | Only if asked |
| **`neutron task rm`** | YAML key deletion isn't worth a command | Only if asked |
| **Generalized runner subsuming existing commands** | Disruptive to stable plugin API | 2.0 if ever |
| **Task-scoped plugins** | Plugins own actions; tasks are user-owned | 2.0 architectural question |
| **Turborepo/Nx migration tooling** | `neutron init` starts from scratch | Community plugin if ever |
| **Windows-first support** | Darwin + Linux only; WSL is the Windows story | Community-maintained |
| **Custom versioning schemes** | SemVer + release-please is working | Never unless a real case |

### Architecture policies re-committed through 1.0

- No new runtime deps without discussion.
- No CLI framework (commander/yargs/oclif). `~30-line router in `src/bin.ts` stays.
- No pre-commit auto-formatting. Checks only.
- `process.exit` only in `src/bin.ts`.
- No Bun-specific APIs in `src/`; Node 20+ for distribution.
- `neutron` owns git hooks. No Husky, no commitlint.
- Bundled tools stay at oxlint + oxfmt. Other ecosystems rely on user/CI toolchains.

### Explicit UX non-promises

- **Not stable:** spinners, colors, ASCII banner, output phrasing. `--json` output is.
- **Not wire-stable:** `.neutron/cache/` layout. Internal; bumps to v2 cache format are valid between minors (must invalidate gracefully).
- **Not version-pinned:** bundled oxlint/oxfmt majors do not gate neutron majors.
